----------
A code path in payment-api v2.8.1 checkout transaction handler failed to call
connection.commit() when a downstream Redis cache check returned a timeout.
The connection was held open (idle_in_transaction) until HikariPool acquisition
timeout fired, cascading into full pool exhaustion.

RESOLUTION
----------
Terminated 17 idle_in_transaction connections from pg_stat_activity.

Patch deployed in v2.8.2: added finally{connection.close()} block in checkout handler.

CONTRIBUTING FACTORS
--------------------
- No alerting on idle_in_transaction count (threshold alert added post-incident)
- HikariPool acquisition timeout set to 30s — long enough to exhaust pool before alert fired
- Redis cache timeout (500ms) not handled gracefully in checkout transaction path

FOLLOW-UP ACTIONS
-----------------
1. Add Prometheus alert: idle_in_transaction > 5 sustained > 2 minutes [DONE]
2. Reduce HikariPool acquisition timeout from 30s to 10s [IN PROGRESS]
3. Add connection.close() in finally block for all checkout paths [DONE — v2.8.2]
4. Load test new checkout path under Redis timeout simulation [SCHEDULED]

----------

malformed messages route directly to dlq.prices without crashing the consumer.
