ENVIRONMENT: Production Architecture Overview
=============================================
Platform: alpha-capital | Region: us-east-1 | Cluster: gke-prod-us-east-1
Last Updated: 2026-01-10 | Owner: Platform Team

SERVICES
--------
nginx-gateway         — Ingress, load balancer, TLS termination. Handles all external traffic.
payment-api           — Core payment processing. Calls postgres-primary (HikariPool) and redis-cache.
order-api             — Order management. Calls payment-api, postgres-primary, kafka-broker.
auth-service          — JWT issuance and validation. JWK endpoint: https://auth.prod.internal/.well-known/jwks.json
risk-engine           — Real-time risk scoring for orders. Consumes risk.signals Kafka topic.
market-data-consumer  — Consumes external price feed via Kafka topic=prices. Publishes to redis-cache.
redis-cache           — Shared cache for pricing data and session state.
kafka-broker          — Message bus. Topics: prices, orders, risk.signals, retry.prices, dlq.prices.
postgres-primary      — Primary PostgreSQL 14 database. 2 read replicas in same AZ.
node-agent            — DaemonSet. Node metrics (CPU, memory, disk), log collection, logrotate.
deploy-controller     — Controls rollouts, canary deployments, rollback triggers.

REPLICA COUNTS (prod, normal operation)
----------------------------------------
payment-api:          4 replicas (HPA: min=4, max=12, trigger: cpu>60%)
order-api:            3 replicas (HPA: min=3, max=8)
auth-service:         2 replicas (HPA: min=2, max=6)
risk-engine:          2 replicas (HPA: min=2, max=6)
market-data-consumer: 3 replicas (HPA: min=3, max=8, trigger: kafka-lag>50000)
nginx-gateway:        2 replicas
postgres-primary:     1 primary + 2 read replicas
redis-cache:          3-node cluster
kafka-broker:         3-node cluster, replication-factor=2

DATABASE
---------
Engine:          PostgreSQL 14.8
Instance type:   n2-highmem-8 (GCP)
max_connections: 200
Storage:         500GB SSD persistent disk (PVC)
Backups:         Daily full, 6-hour WAL shipping to GCS
Connection pool: HikariPool per service pod
  payment-api max per pod:  25 connections
  order-api max per pod:    15 connections
  auth-service max per pod: 5 connections

KAFKA
------
Version:          3.5.1
Broker count:     3
Topic: prices     — 8 partitions, replication=2, retention=24h
Topic: orders     — 4 partitions, replication=2, retention=72h
Topic: retry.prices — 8 partitions, replication=2, retention=6h
Topic: dlq.prices   — 1 partition, replication=2, retention=7 days
Consumer group: md-consumer (market-data-consumer, 3 consumers = 3 partitions each)

NETWORKING
-----------

NODE POOL
----------

OS disk:      100GB SSD
Ephemeral log storage per node: ~20GB usable
Node IPs:     ip-10-0-1-10 through ip-10-0-1-19 range (us-east-1)

MONITORING AND ALERTING
------------------------
Metrics:    Prometheus + Grafana (internal: grafana.prod.internal)
Alerting:   PagerDuty, #devops-incidents Slack channel
Log aggregation: Cloud Logging (GCP), 30-day retention
Tracing:    Jaeger (trace_id format: trc-XXXXXXXXXX)
Key dashboards:
  - PostgreSQL Overview (ID: 101)
  - API Gateway Request Rate (ID: 102)
  - Kafka Consumer Lag (ID: 103)
  - Pod Resource Usage (ID: 104)
  - Node Disk and Memory (ID: 105)

DEPLOYMENT
-----------
CI/CD: GitHub Actions → GCR image push → deploy-controller applies to GKE
Canary: default 10% traffic, 5-minute bake time, error budget threshold
Rollback trigger: error_budget_burn=high (auto-flag) or manual SRE/release-manager decision
Deploy schedule: weekdays 10:00-16:00 UTC (avoid market open 09:30 EST = 14:30 UTC)
Freeze periods: last 2 business days of each month, major market events
