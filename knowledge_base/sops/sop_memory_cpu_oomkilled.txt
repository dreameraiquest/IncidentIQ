SOP: Memory / CPU Saturation — OOMKilled and CPU Throttling
============================================================
Stack: payment-api / risk-engine / order-api | GKE us-east-1 | env=prod
Category: Memory/CPU | Severity: P1 | Owner: SRE Team

TRIGGER CONDITIONS
------------------
- kubectl describe pod shows OOMKilled exit code 137 on any prod pod
- logger=kubelet: "OOMKilled" or "memory pressure" on prod nodes
- payment-api or risk-engine pod restart count > 3 in 10 minutes
- CPU throttling_ratio > 0.5 sustained > 5 minutes on payment-api or risk-engine

IMMEDIATE TRIAGE (0-5 minutes)
--------------------------------
1. Check for OOMKilled pods:
   kubectl get pods -n prod | grep -v Running
   kubectl describe pod <oom-pod> -n prod | grep -A10 "Last State\|OOMKilled\|Exit Code"

2. Check current memory usage vs limits for critical pods:
   kubectl top pods -n prod --sort-by=memory | head -10
   kubectl describe pod payment-api-<pod> -n prod | grep -A4 "Limits\|Requests"

3. Check restart history:
   kubectl get pods -n prod -o wide | awk '{print $4, $1}' | sort -rn | head -10

4. Check if restart correlates with recent deploy:
   kubectl rollout history deployment/payment-api -n prod
   kubectl rollout history deployment/risk-engine -n prod

ROOT CAUSE PATTERNS
--------------------
Pattern A — Memory leak in new version (most common after deploy):
  Signal: OOMKilled starts after deploy-controller "rollout started" log
  Cause: v2.9.0 introduced unbounded cache or connection accumulation
  Evidence: memory usage climbing linearly before OOMKilled

Pattern B — Traffic spike exceeding limits:
  Signal: OOMKilled across multiple pods simultaneously, no recent deploy
  Cause: peak load (market open, flash sale) exceeds pod memory limits
  Evidence: request rate spike in nginx-gateway logs before OOM

Pattern C — JVM heap misconfiguration:
  Signal: risk-engine or payment-api OOMKilled with GC overhead logs before crash
  Check: kubectl exec -it <pod> -n prod -- \
  curl -s localhost:8080/actuator/metrics/jvm.memory.max | jq

REMEDIATION
-----------
Step 1 — No approval needed (preserve evidence first):
  Capture heap dump before pod restarts again:
    kubectl exec -it payment-api-<pod> -n prod -- \
    curl -X POST localhost:8080/actuator/heapdump -o /tmp/heap-$(date +%s).hprof

Step 2 — If Pattern A (bad deploy — release-manager approval):
  kubectl rollout undo deployment/payment-api -n prod
  kubectl rollout status deployment/payment-api -n prod

Step 3 — If Pattern B (traffic spike — SRE approval):
  Horizontal scale to distribute load:
    kubectl scale deployment/payment-api --replicas=8 -n prod
    kubectl scale deployment/risk-engine --replicas=4 -n prod

Step 4 — Temporary memory limit increase (SRE + service-owner approval):
  kubectl set resources deployment/payment-api \
  --limits=memory=2Gi --requests=memory=1Gi -n prod
  ⚠️ Profile first — blindly increasing limits delays finding the real leak

VALIDATION
----------
✅ OOMKilled events stop — kubectl get pods -n prod shows all Running
✅ Restart count stabilises at 0 new restarts over 10 minutes
✅ Memory usage stays below 80% of limit under normal load
✅ CPU throttling_ratio drops below 0.1

SAFETY RULES
------------
❌ Do NOT repeatedly restart OOMKilled pods without capturing diagnostics first
❌ Do NOT increase limits without profiling — masks memory leak
❌ Do NOT scale pods that are OOMKilling — they will OOM at scale too
✅ Always capture heap dump or memory snapshot before restarting
✅ Check for deploy correlation before assuming traffic is the cause

ESCALATION
----------
0-5 min:  On-call SRE — capture diagnostics immediately
5-15 min: Service owner (payment-api or risk-engine) to review heap dump
15+ min:  Engineering manager if OOMKill loop is causing payment outage
