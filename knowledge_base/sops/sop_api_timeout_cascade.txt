SOP: API Timeout and Upstream Latency Cascade
=============================================
Stack: nginx-gateway / payment-api / order-api | GKE us-east-1 | env=prod
Category: API timeout | Severity: P2 | Owner: Platform Team

TRIGGER CONDITIONS
------------------
- logger=nginx: "upstream timed out while reading response header from upstream request=/api/payments/authorize"
- logger=nginx: "HTTP 504 generated for /api/payments/authorize after upstream timeout"
- nginx-gateway status=504 on /api/payments/authorize sustained > 1 minute
- p95 latency on payment-api /api/payments/authorize exceeds 10 seconds

IMMEDIATE TRIAGE (0-5 minutes)
--------------------------------
1. Confirm 504 scope — is it one endpoint or all?
   kubectl logs -l app=nginx-gateway -n prod --since=5m | grep 504 | awk '{print $7}' | sort | uniq -c

2. Check upstream response time from nginx perspective:
   kubectl logs -l app=nginx-gateway -n prod --since=5m | \
   grep "upstream timed out" | grep -oP 'duration_ms":\K[0-9]+'

3. Check payment-api pod health directly (bypass nginx):
   kubectl exec -it nginx-gateway-<pod> -n prod -- \
   curl -w "%{time_total}" -s -o /dev/null http://payment-api.prod.svc/health

4. Check if circuit breaker is already open:
   kubectl exec -it payment-api-<pod> -n prod -- \
   curl -s localhost:8080/actuator/circuitbreakers | jq '.circuitBreakers[].state'

ROOT CAUSE PATTERNS
--------------------
Pattern A — DB pool exhaustion causing upstream hold (most common — check sop_database_connection_pool.txt):
  Signal: 504 on /api/payments/authorize + HikariPool timeout in same window
  Cause: payment-api holding connections waiting for DB, nginx times out waiting for payment-api
  This is a downstream symptom — the root cause is Database category

Pattern B — payment-api GC pause causing latency spike:
  Signal: 504 spike without DB errors, payment-api pod CPU normal but latency high
  Check: kubectl logs payment-api-<pod> -n prod --since=10m | grep -i "gc pause\|GC overhead"

Pattern C — nginx timeout configuration mismatch after deploy:
  Signal: 504 started exactly at deployment of nginx-gateway
  Check: kubectl get configmap nginx-config -n prod -o yaml | grep proxy_read_timeout

REMEDIATION
-----------
Step 1 — No approval needed:
  Check if circuit breaker can absorb load while upstream recovers:
    kubectl exec -it payment-api-<pod> -n prod -- \
    curl -X POST localhost:8080/actuator/circuitbreakers/payment-db/force-open
  This fails fast instead of holding connections for 30s timeout

Step 2 — If Pattern A (DB is root cause):
  Follow sop_database_connection_pool.txt — fix DB first, 504s will self-resolve

Step 3 — If Pattern B (GC pause — SRE approval):
  Trigger rolling restart to clear GC state:
    kubectl rollout restart deployment/payment-api -n prod
  ⚠️ Only if DB is confirmed healthy — restarting during DB exhaustion makes it worse

Step 4 — Temporary timeout adjustment (platform approval):
  If upstream is slow but healthy, temporary increase prevents false 504s:
    kubectl patch configmap nginx-config -n prod \
    --patch '{"data":{"proxy_read_timeout":"120s"}}'
    kubectl rollout restart deployment/nginx-gateway -n prod
  ⚠️ Revert after upstream latency normalises — this masks problems if left permanently

VALIDATION
----------
✅ nginx-gateway 504 rate drops to < 0.01%
✅ payment-api /api/payments/authorize p95 latency < 2 seconds
✅ circuit breaker state returns to CLOSED
✅ No new "upstream timed out" entries in nginx-gateway logs

SAFETY RULES
------------
❌ Do NOT increase timeouts as the permanent fix — find why upstream is slow
❌ Do NOT restart payment-api if DB pool exhaustion is the actual root cause
✅ Always check if 504 is symptom of DB issue before treating it as the root cause
✅ Document which endpoint is affected — helps narrow RCA to one upstream service

ESCALATION
----------
0-5 min:  On-call SRE
5-15 min: payment-api owner + platform team if DB not the cause
15+ min:  Engineering manager if payment processing is customer-impacting
